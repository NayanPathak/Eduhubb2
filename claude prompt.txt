# EduHub – Angular Frontend Build Prompt

Use this prompt to generate the complete Angular frontend for the **EduHub** application.

---

## 0. Hard Constraints (do not violate)

1. **Angular version:** Angular 16, **class-based components only** (no standalone/functional components, no signals-based syntax).
2. **Forms:** **Template-driven forms only** (`FormsModule`, `ngModel`, `#form="ngForm"`). Do **NOT** use Reactive Forms (`FormGroup`, `FormBuilder`, `formControlName`).
3. **Styling:** Tailwind CSS **v3.4.17** only. No Bootstrap, no Angular Material, no inline `<style>` blocks unless unavoidable.
4. **File structure per component:** Every component must have **3 separate files**: `component-name.component.ts`, `component-name.component.html`, `component-name.component.css` (or scss, kept minimal since Tailwind handles styling). **Do not use inline templates or inline styles** (`template:` / `styles:` inside the `@Component` decorator is forbidden).
5. **Models are FROZEN.** Use the following model files exactly as given below, with **exact property names and casing**. Do not rename, re-case, add, or remove any property (e.g., `CourseId` must remain `CourseId`, not `courseId` or `course_id`). Every component/service must bind to these exact property names in both TypeScript and HTML (e.g., `course.CourseId`, `[(ngModel)]="course.Title"`).
6. Do not modify `course.model.ts`, `enrollment.model.ts`, `feedback.model.ts`, `login.model.ts`, `material.model.ts`, `user.model.ts` — treat them as already final and existing in `src/app/models/`.
7. Angular app runs on **port 8081**; backend (.NET) runs on **port 8080**. Base API URL: `http://localhost:8080`.
8. JWT token must be stored in `localStorage`, sent as `Authorization: Bearer <token>` header on every protected request.
9. Routing must be protected with an **AuthGuard** implementing `CanActivate`. Unauthenticated users can only access `/login` and `/register`. Login page is the default/first route.
10. Role-based route protection: Educator and Student have different accessible routes/components (see Section 4).

## 4. Routing & Guards

- `''` and `'login'` → `LoginComponent` (default/landing route)
- `'register'` → `RegisterComponent`
- `'home'` → `HomeComponent` (protected)
- Educator routes (protected + role check for Educator): course list/create/edit, view enrollments per course, upload materials, view feedback list.
- Student routes (protected + role check for Student): `studentviewcourse` (browse + enroll), enrolled courses, course materials, feedback (view/post).
- `'**'` → `ErrorPageComponent`

`AuthGuard` (`CanActivate`):
- Reads token from `localStorage`.
- If no token → redirect to `/login`.
- Optionally validate role against route data (`route.data['role']`) and redirect to an error/forbidden view if mismatched.

`AuthInterceptor` (`HttpInterceptor`):
- Attaches `Authorization: Bearer <token>` header to every outgoing request (except login/register) using the token from `localStorage`.

Logout: clear `localStorage` and navigate to `/login`.

## 5. Services (methods, endpoints, and exact request/response typing)

All services declare `public apiUrl = 'http://localhost:8080';` and build headers using the stored JWT token, prefixed with `'Bearer '`.

**auth.service.ts**
- `register(user: RegisterRequest): Observable<RegisterResponse>` → `POST /api/register`
- `login(login: Login): Observable<LoginResponse>` → `POST /api/login`; on success, store `token`, `role`, `UserId` in `localStorage`.
- `logout(): void` → clear `localStorage`.
- Helper getters: `getToken()`, `getRole()`, `getUserId()`.

**course.service.ts**
- `getAllCourses(): Observable<Course[]>` → `GET /api/course`
- `getCourseById(id: number): Observable<Course>` → `GET /api/course/{courseId}`
- `addCourse(course: Course): Observable<Course>` → `POST /api/course`
- `updateCourse(id: number, course: Course): Observable<Course>` → `PUT /api/course/{courseId}`
- `deleteCourse(courseId: number): Observable<void>` → `DELETE /api/course/{courseId}`

**enrollment.service.ts**
- `getAllEnrollments(): Observable<Enrollment[]>` → `GET /api/enrollment`
- `enrollCourse(courseId: string, userId: string): Observable<Enrollment[]>` → `POST /api/enrollment`
- `getEnrolledCourses(userId: string): Observable<Enrollment[]>` → `GET /api/enrollment/user/{userId}`
- `updateEnrollmentStatus(id: string, enrollment: Enrollment): Observable<Enrollment>` → `PUT /api/enrollment/{enrollmentId}`
- `unenrollCourse(enrollmentId: string): Observable<Enrollment>` → `DELETE /api/enrollment/{enrollmentId}`
- `getEnrollmentsByCourse(courseId: string): Observable<Enrollment[]>` → `GET /api/enrollment/course/{courseId}`

**material.service.ts**
- `getMaterialsByCourseId(courseId: string): Observable<Material[]>` → `GET /api/material/course/{courseId}`
- `addMaterial(material: Material): Observable<Material>` → `POST /api/material`
- `deleteMaterial(materialId: number): Observable<any>` → `DELETE /api/material/{materialId}`

**feedback.service.ts**
- `sendFeedback(feedback: Feedback): Observable<Feedback>` → `POST /api/feedback`
- `getAllFeedbacksByUserId(userId: string): Observable<Feedback[]>` → `GET /api/feedback/{userId}`
- `deleteFeedback(feedbackId: string): Observable<void>` → `DELETE /api/feedback/{feedbackId}`
- `getFeedbacks(): Observable<Feedback[]>` → `GET /api/feedback`

Bind every field in forms/tables to the exact model property names (`CourseId`, `Title`, `Description`, `CourseStartDate`, `CourseEndDate`, `Category`, `Level`, `EnrollmentId`, `UserId`, `Status`, `MaterialId`, `URL`, `UploadDate`, `ContentType`, `FeedbackId`, `FeedbackText`, `Date`, `Email`, `Password`, `Username`, `MobileNumber`, `UserRole`).

## 6. Components — behavior spec

For **each** component below, generate:
- A class-based `@Component` with a **separate `.html` template file** and **separate `.css` file** (use `templateUrl` / `styleUrls`, never inline).
- Template-driven form (`#xForm="ngForm"`, `[(ngModel)]`, `required`, `pattern`, `minlength` etc. as HTML5/Angular validators) wherever user input is collected.
- Tailwind utility classes for all layout/styling (cards, tables, buttons, navbar, forms, alerts).
- Inline validation messages shown conditionally via `*ngIf="field.invalid && field.touched"`.

1. **LoginComponent** – email + password template-driven form; on submit calls `AuthService.login`; on success routes to `/home` based on role; shows "Invalid email"/"Invalid password" server errors.

2. **RegisterComponent** – username, email, mobile number, password, role select (Educator/Student); on submit calls `AuthService.register`; on success redirect to login with success message.

3. **NavbarComponent** – shows different links depending on role (Educator: Courses, Enrollments, Feedback; Student: Courses, Enrolled Courses, Feedback). Hovering "Courses" reveals a submenu with **"View Courses"** (required id per SRS). Includes Logout button.

4. **HomeComponent** – landing dashboard after login, simple welcome content, differs slightly per role (optional).

5. **EducatorCourseListComponent** – lists all courses (`getAllCourses`), with Edit/Delete actions and a link to add a new course; delete confirmation before calling `deleteCourse`.

6. **CourseFormComponent** – template-driven form for Title, Description, CourseStartDate, CourseEndDate, Category, Level; reused for both Add (`addCourse`) and Edit (`updateCourse`) via a route param check.

7. **CourseEnrollmentsComponent** – for a given `CourseId`, lists enrollments (`getEnrollmentsByCourse`) with student info and status (Accepted/Rejected/Pending); allow educator to update status via `updateEnrollmentStatus`.

8. **CourseMaterialsComponent** (educator side) – lists materials for a course (`getMaterialsByCourseId`), form to add a new material (Title, Description, URL, ContentType), and delete action (`deleteMaterial`).

9. **EducatorFeedbackListComponent** – lists all feedback (`getFeedbacks`), read-only view for educators.

10. **StudentviewcourseComponent** *(id must remain exactly `studentviewcourse` per SRS)* – lists all available courses for students to browse; enroll button calls `enrollCourse`.

11. **StudentEnrolledCoursesComponent** – lists the logged-in student's enrollments (`getEnrolledCourses`), with unenroll action (`unenrollCourse`) and status display.

12. **StudentCourseMaterialsComponent** – shows materials for a course the student is enrolled in (`getMaterialsByCourseId`), read-only.

13. **StudentFeedbackComponent** – template-driven form to submit feedback (`sendFeedback`); list of own past feedback (`getAllFeedbacksByUserId`) with delete option (`deleteFeedback`).

14. **ErrorPageComponent** – generic 404/500-style fallback page with a friendly message and a link back to `/home`.

## 7. Validation Rules (client-side, HTML5 + template-driven)

- Email: `required`, `pattern` for basic email format.
- Password: `required`, `minlength="6"`.
- Mobile number: `required`, `pattern="^[0-9]{10}$"`.
- Course dates: `required`; optionally validate `CourseEndDate >= CourseStartDate` in the component class.
- All text fields (`Title`, `Description`, `Category`, `Level`, `FeedbackText`, `URL`): `required`.
- Disable submit button while the form is invalid (`[disabled]="form.invalid"`).
